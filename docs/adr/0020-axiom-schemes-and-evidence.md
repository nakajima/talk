# ADR 0020: Axiom schemes, givens/wanteds, and evidence

## Status

Proposed.

## Context

Talk already has many pieces of an OutsideIn-style qualified type system:

- predicates such as protocol conformance, type equality, row equality, effect
  equality, and member predicates;
- generated constraints from expressions;
- local assumptions from `where` clauses, generic bounds, GADT refinements, and
  existential unpacking;
- top-level declarations that behave like reusable conformance rules;
- associated type projections;
- protocol witnesses, defaults, derived conformances, and existential witness
  tables;
- lowering that needs to know exactly which witness was selected.

The implementation currently spreads this across several mechanisms:

- nominal conformance maps;
- protocol bounds and superprotocol closure helpers;
- associated-type projection reduction through catalog lookup;
- `MemberResolution::ViaConformance` plus witness symbols;
- per-node instantiation tables;
- protocol default handling;
- derived conformance special cases;
- lowerer-side witness rediscovery.

The motivating feature is:

```talk
extend Iterator: Into<Array<Element>> {
    consuming func into() -> Array<Element> { ... }
}
```

Semantically, this is a top-level axiom scheme:

```text
forall Self. Self: Iterator => Self: Into<Array<<Self as Iterator>.Element>>
```

Existing conformances have the same shape:

```talk
extend Array<Element>: Iterable { ... }
```

is:

```text
forall Element. true => Array<Element>: Iterable
```

and:

```talk
extend Array<Element>: Showable where Element: Showable { ... }
```

is:

```text
forall Element. Element: Showable => Array<Element>: Showable
```

The desired architecture is therefore not a special "blanket conformance" path.
It is a single OutsideIn-aligned model of:

```text
top-level axiom schemes + local givens + generated wanteds + evidence
```

## Research basis

This ADR follows the terminology and split in Vytiniotis, Peyton Jones,
Schrijvers, and Sulzmann, "OutsideIn(X): Modular type inference with local
assumptions" (JFP 2011):

- Source typing generates wanted constraints.
- Type signatures, pattern matches, and other local assumptions introduce given
  constraints.
- Top-level reusable rules such as Haskell instances are axiom schemes.
- The solver is parameterized by an underlying constraint domain `X`.
- The simplifier must be sound and guess-free.
- The implementation elaborates programs with evidence terms.

The paper's type-class instance example:

```haskell
instance Eq a => Eq [a] where ...
```

is represented as a top-level axiom scheme:

```text
forall a. Eq a => Eq [a]
```

Talk protocol conformance declarations should use the same conceptual model.

The paper also warns about exactly the failure modes relevant to Talk:

- applying top-level axioms can introduce new wanted constraints;
- givens and wanteds must remain distinct because wanteds may contain touchable
  inference variables;
- givens must not be polluted with evidence that depends on solving wanteds;
- overlapping axiom schemes create nondeterminism;
- recursive axiom schemes threaten termination;
- evidence terms should record the proof selected by the solver.

## Terminology

Use the OutsideIn terms precisely.

### Axiom scheme

A top-level reusable rule. It is global and can be instantiated many times.

Shape:

```text
forall params. premises => conclusion
```

Examples:

```text
forall A. A: Showable => Array<A>: Showable
forall Self. Self: Iterator => Self: Into<Array<<Self as Iterator>.Element>>
```

Axioms are not local evidence. They are rules for constructing evidence when
premises are proven.

### Given

A local assumption currently in scope, with evidence already available.

Examples:

```talk
func f<T>(x: T) where T: Showable { ... }
```

Inside `f`:

```text
T: Showable
```

is a given.

A protocol extension body also has a given for its head protocol:

```talk
extend Iterator { ... }
```

Inside the body:

```text
Self: Iterator
```

is a given.

### Wanted

An obligation generated by source code.

Examples:

```talk
x.show()
```

generates a wanted such as:

```text
typeof(x): Showable
```

The solver discharges wanteds using local givens, top-level axiom schemes, and the
built-in constraint theory.

### Evidence

A proof term explaining how a wanted was solved. Evidence may be erased or
monomorphized later, but the typed program must record it so lowering does not
repeat witness selection.

## Decision

Represent protocol conformances, superprotocol entailments, protocol-extension
conformances, derived conformances, and related reusable entailments as top-level
axiom schemes.

The checker solves generated wanteds in the presence of local givens and top-level
axiom schemes. Successful solving produces evidence. The typed program stores
that evidence. Lowering consumes evidence and must not rediscover witnesses by
asking the catalog again.

Do not implement a separate blanket-conformance subsystem.

Do not treat top-level axiom schemes as local givens. They are global rules that
may be used to solve wanteds by producing new wanted premises.

## Non-goals

This ADR does not add performance indexes.

The initial implementation should use the simplest correct architecture:

```text
Vec<AxiomScheme>
```

Candidate search may linearly scan axiom schemes. This is deliberate. Indexes can
be introduced later behind the same semantic API only if profiling proves they are
needed. They must not become semantic sources of truth.

This ADR also does not add:

- specialization;
- overlap priority;
- incoherent instances;
- runtime dictionary search;
- arbitrary extension heads such as `extend<T: Iterator> T`.

## Core data model

### Predicate

Talk already has a predicate language. Keep it as the language of givens,
wanteds, axiom premises, and axiom conclusions.

Conformance predicates use protocol references:

```rust
Predicate::Conforms {
    ty: Ty,
    protocol: ProtocolRef,
}
```

Protocol references include protocol arguments. Associated type equalities remain
separate predicates or equations carried by selected evidence.

### AxiomScheme

Introduce one semantic representation for top-level reusable entailments:

```rust
pub struct AxiomScheme {
    pub id: AxiomSchemeId,
    pub params: Vec<Symbol>,
    pub premises: Vec<Predicate>,
    pub conclusion: Predicate,
    pub evidence: EvidenceTemplate,
    pub origin: NodeID,
}
```

For v1, `conclusion` should be an evidence-bearing predicate, primarily protocol
conformance. Later ADRs may decide whether other predicate forms should be
admitted as axiom conclusions.

A simple supporting shape:

```rust
pub struct AxiomSchemeId(pub u32);

pub enum EvidenceTemplate {
    Conformance {
        witnesses: WitnessSet,
        assoc: Vec<AssocEquation>,
    },
    Superprotocol,
    Derived(DerivedEvidenceKind),
}

pub struct WitnessSet {
    pub methods: FxHashMap<String, Symbol>,
}

pub struct AssocEquation {
    pub assoc: Symbol,
    pub ty: Ty,
}
```

The exact Rust names may change. The important architectural point is that the
semantic rule is the axiom scheme, not a nominal-head catalog entry.

### Given evidence

Local givens need evidence handles, not just predicates:

```rust
pub struct Given {
    pub predicate: Predicate,
    pub evidence: Evidence,
}
```

A function `where T: Showable` introduces a given with evidence supplied by the
caller's obligation. A GADT branch introduces equality evidence. An existential
unpack introduces evidence from the package.

### Wanted

A wanted is an originated predicate:

```rust
pub struct Wanted {
    pub predicate: Predicate,
    pub origin: CtOrigin,
}
```

The solver may solve it, leave it residual where allowed, or report a diagnostic.

### Evidence

Solving a wanted produces evidence:

```rust
pub enum Evidence {
    Given {
        given: GivenId,
    },
    Axiom {
        axiom: AxiomSchemeId,
        substitution: Vec<(Symbol, Ty)>,
        premises: Vec<Evidence>,
    },
    Derived {
        kind: DerivedEvidenceKind,
        predicate: Predicate,
        premises: Vec<Evidence>,
    },
    Error,
}
```

Evidence is the typed-program boundary between checking and lowering.

## Underlying constraint domain X

Not every constraint is an axiom scheme.

OutsideIn separates the generic implication solver from the underlying domain `X`.
For Talk, `X` includes specialized machinery for:

- type equality and unification;
- associated-type/type-family-like projection equalities;
- row equality;
- effect-row equality;
- borrow-shaped type compatibility;
- canonicalization and interaction between flat constraints.

These are solver theories, not reusable user axiom schemes.

Axiom schemes may produce or require predicates that the domain solver then
simplifies. For example, applying a conformance axiom may create a new wanted
conformance premise and equality obligations for associated types.

## Solver behavior

To solve a wanted predicate:

```text
W
```

1. Normalize/canonicalize `W` through the built-in constraint domain where
   applicable.
2. Try to discharge `W` from local givens without applying top-level axiom
   schemes to the givens.
3. If not solved by givens, linearly scan top-level axiom schemes.
4. For each axiom, try to match its conclusion against `W`.
5. Matching produces a substitution for the axiom parameters.
6. Substitute the axiom premises.
7. Recursively solve each substituted premise as a new wanted.
8. If all premises solve, the axiom is viable and produces `Evidence::Axiom`.
9. If exactly one axiom is viable, use that evidence.
10. If no axiom is viable, leave the wanted residual where allowed or report a
    diagnostic.
11. If multiple axioms are viable, report overlapping or ambiguous evidence.

Axiom order is not meaningful. The solver must not choose the first viable axiom.
It must prove uniqueness.

## Given/wanted discipline

Maintain the OutsideIn distinction strictly:

- Givens are local evidence.
- Wanteds are obligations.
- Axiom schemes are global rules.

Applying an axiom to a wanted may create new wanteds:

```text
axiom:  A: Showable => Array<A>: Showable
wanted: Array<T>: Showable
new wanted: T: Showable
```

Do not turn wanted-derived facts into givens.

Do not expand local givens through top-level axiom schemes as a normal solving
step. If a given directly entails a wanted, use the given evidence. If using an
axiom would also solve the same wanted, the selected evidence must remain
coherent and deterministic.

This rule exists to avoid the nondeterminism described in OutsideIn's top-level
reaction discussion.

## Coherence and overlap

For every fully known wanted predicate, there must be at most one valid evidence
term.

V1 rules:

- If two axiom schemes can both prove the same wanted predicate, reject the
  program.
- There is no specialization relation.
- There is no priority based on declaration order.
- Imported axiom schemes participate in the same overlap checks as local axioms.
- If the checker cannot prove two axiom schemes are disjoint, reject the later
  declaration with an overlap diagnostic.

This is conservative but simple. It preserves the invariant that typed code has one
canonical proof for each solved wanted.

## Termination

Axiom-scheme solving must detect recursive entailment cycles explicitly.

Maintain a goal stack during axiom reaction:

```text
normalized wanted predicate + relevant givens fingerprint
```

If solving an axiom premise re-enters the same goal without progress, report a
recursive conformance/axiom diagnostic naming the involved axioms and predicate.

The hard solver budget remains a defensive fail-closed guard, not a normal
user-facing mechanism.

## Associated types

Associated type projections reduce through evidence.

A projection:

```text
<T as P<Args...>>::Assoc
```

is reduced by first obtaining evidence for:

```text
T: P<Args...>
```

Then the selected evidence supplies the associated-type equation.

For:

```talk
extend Iterator: Into<Array<Element>> { ... }
```

if `Into` has an associated `Target`, selected evidence gives:

```text
<Self as Into>.Target == Array<<Self as Iterator>.Element>
```

This reduction must not be a separate catalog lookup independent of the evidence
selected for `Self: Into<...>`.

## Member lookup

Member lookup through a protocol requirement should produce an evidence-backed
call target.

Current output roughly says:

```rust
MemberResolution::ViaConformance {
    protocol,
    witness,
}
```

The new output should represent:

```text
requirement + selected evidence + method substitution
```

For example:

```talk
iter.into()
```

should type check by proving:

```text
typeof(iter): Into<Array<Element>>
```

and the call node should store evidence for that proof. Lowering then compiles the
witness selected by that evidence.

This replaces lowerer-side witness rediscovery.

## Superprotocols

A superprotocol declaration generates an axiom scheme.

```talk
protocol Comparable<RHS>: Equatable<RHS> { ... }
```

becomes:

```text
forall Self RHS. Self: Comparable<RHS> => Self: Equatable<RHS>
```

Superprotocol closure should be a consequence of axiom solving, not a separate
closure helper used by some paths but not others.

## Associated type bounds

An associated type bound generates an axiom scheme or evidence-bearing
entailment attached to the protocol evidence.

```talk
protocol Iterator {
    associated Element: Showable
}
```

means:

```text
forall Self. Self: Iterator => <Self as Iterator>.Element: Showable
```

Whether this is represented as a separate axiom scheme or as an entailment inside
the `Iterator` evidence template is an implementation choice. It must obey the
same given/wanted discipline and produce evidence when used.

## Protocol defaults

Protocol defaults are evidence templates, not a separate dispatch mechanism.

When an axiom scheme proves a conformance and no explicit witness is supplied for
a requirement, the evidence template may use a default witness if its own premises
are satisfied.

Default selection is recorded in evidence. Lowering consumes that evidence.

## Derived conformances

Derived conformances are synthetic axiom schemes.

For example, derived `Showable` for a struct is modeled as:

```text
fields Showable => Struct: Showable
```

with a compiler-synthesized evidence template.

This keeps derivation special only at axiom generation and witness synthesis, not
in conformance solving.

## Existentials

Packing an existential requires evidence for the payload conformance.

A typed existential pack should carry:

```text
payload expression + payload type + evidence for payload type: protocol
```

Lowering builds the runtime witness table from evidence. It should not repeat
conformance lookup.

## Conversion protocols

`Into` should become a protocol-argument protocol:

```talk
public protocol Into<Target> {
    consuming func into() -> Target
}
```

The target is an input to conformance selection, not an output uniquely determined
by `Self`. This follows ADR 0016 and permits one source type to support multiple
conversions.

Then the motivating axiom scheme is natural:

```talk
extend Iterator: Into<Array<Element>> {
    consuming func into() -> Array<Element> { ... }
}
```

If `Into` remains temporarily associated-type based, evidence can still model it,
but the long-term library design should migrate `Into` and `From` to protocol
arguments.

## Source syntax

Accepted v1 syntax:

```talk
extend Iterator: Into<Array<Element>> {
    consuming func into() -> Array<Element> { ... }
}

extend Iterator: Showable where Element: Showable {
    func show() -> String { ... }
}
```

In a protocol-extension conformance:

- `Self` is the unknown conforming type.
- The head protocol's associated names are in scope.
- The target conformance may mention `Self`, protocol arguments, and associated
  projections from the head protocol.

Rejected for v1:

```talk
extend<T: Iterator> T: Into<Array<T.Element>> { ... }
```

Diagnostic should say this is an arbitrary blanket extension head and suggest a
protocol-extension conformance when the head predicate is a single protocol.

## Implementation plan

1. Replace the conformance-clause draft terminology with OutsideIn terminology:
   axiom schemes, givens, wanteds, and evidence.
2. Introduce `AxiomSchemeId`, `AxiomScheme`, `Given`, `Wanted`, and `Evidence`
   in the type layer.
3. Generate axiom schemes for existing nominal conformances without changing
   source behavior.
4. Change conformance solving to solve wanted predicates against local givens and
   top-level axiom schemes, returning evidence.
5. Preserve existing diagnostics and tests for nominal and conditional
   conformances.
6. Move associated-type projection reduction to evidence-backed reduction.
7. Replace `MemberResolution::ViaConformance` with an evidence-backed call
   resolution.
8. Represent superprotocols as axiom schemes.
9. Represent associated type bounds as axiom schemes or evidence-template
   entailments.
10. Move protocol defaults into evidence templates.
11. Represent derived conformances as synthetic axiom schemes.
12. Make existential packing carry evidence.
13. Remove lowerer-side witness rediscovery; lowering consumes typed evidence.
14. Add explicit recursive axiom/goal detection.
15. Add conservative overlap checks for axiom schemes.
16. Migrate `Into` and `From` to protocol-argument protocols.
17. Enable protocol-extension axiom schemes, including:

    ```talk
    extend Iterator: Into<Array<Element>> { ... }
    ```

18. Add end-to-end tests for:
    - existing nominal conformances;
    - existing conditional conformances;
    - superprotocol entailment;
    - associated type bounds;
    - protocol defaults;
    - derived `Showable`;
    - existential packing;
    - `Iterator: Into<Array<Element>>`;
    - ambiguous overlapping axioms;
    - recursive axioms with a normal diagnostic;
    - LSP and `talk check` agreeing.

## Consequences

Benefits:

- Terminology and architecture align with OutsideIn(X).
- Nominal, conditional, derived, defaulted, superprotocol, and protocol-extension
  conformances share one axiom/evidence model.
- Protocol-extension `Iterator: Into<Array<Element>>` becomes a natural axiom
  scheme, not a special subsystem.
- Associated types reduce through selected evidence, matching the literature.
- Lowering receives explicit proof terms and stops rediscovering witnesses.
- Coherence and termination become central invariants.

Costs:

- This is larger than adding a one-off `Iterator -> Into<Array<Element>>` path.
- Several current tables and side channels must migrate or disappear.
- The type checker/lowerer boundary changes: typed output must carry evidence.
- Initial axiom lookup is linear by design; performance work is deferred.

## Rejected alternatives

### Call top-level axiom schemes "givens"

Rejected. OutsideIn distinguishes top-level axiom schemes from local givens.
Axioms are reusable rules; givens are local evidence already in scope. Collapsing
these terms obscures the evidence discipline and risks wanted-derived evidence
polluting givens.

### Add a blanket conformance table

Rejected because it preserves two semantic paths: nominal conformances and
blanket conformances. The desired model has one path: top-level axiom schemes.

### Keep witness selection in lowering

Rejected because it duplicates typechecker work and can disagree with the proof
used during checking. The checker should publish evidence; lowering should
consume it.

### Add arbitrary `extend<T: P> T` first

Rejected for v1. It is more general than needed and requires arbitrary type
patterns as extension heads. Protocol-extension axiom schemes are enough for the
motivating use case.

### Allow overlap with declaration-order priority

Rejected. Declaration order must not affect evidence selection.

### Add indexes now

Rejected for now. The cleanest architecture is a list of axiom schemes plus a
single solver path. Indexes are optimization and should wait until correctness and
ownership boundaries are settled.
