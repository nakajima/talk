# 0017 — Structural temp drops and a free-balance verifier

Status: superseded by ADR 0031 (2026-07-13)

Stage checklist (see "Stages" below; the open stages are tracked by
`docs/ownership-soundness-plan.md`):

- Stage 1 (repro tests): bug A's shapes landed with stage 3; bug B's
  shape is now a running both-engine test
  (`stdlib_fs_directory_runner_runs_balanced_on_both_engines`,
  plan track 8.3, landed 2026-07-11).
- Stage 2 (free-balance verifier): **landed** — `src/lambda_g/balance.rs`
  (plan track 6.1), ON by default in cfg(test) at `Driver::lower` since
  the 6.2 accounting sweep (2026-07-12), alongside VM + evaluator leak
  fences on every driver run path.
- Stage 3 (structural temp drops): **landed** — `pending_temp_drops`,
  `note_pending_temp`, and `drain_temp_drops` are deleted; statements
  emit their own `TemporaryEnd` candidates. Bug A is fixed.
- Stage 4 (`lower_embedded_body`): **landed** (2026-07-12, plan track
  7.3) — `StatementScope` bundles the statement-scoped builder fields
  (including `scope_floor`, which makes embedded-body early-exit
  isolation structural rather than classification-neutralized), and
  `lower_embedded_body` swaps it wholesale at both sites. The swap of
  the previously-unmasked fields fixed a latent `block_value_temps`
  leak (a trailing block's value tail inside a block expression
  delivered to the enclosing block's temp — λ_G T-App panic), per this
  ADR's thesis: no per-field handling left to forget.
- Stage 5 (bug B typing fix): **landed** (2026-07-11, plan track 8.3) —
  root cause was solver `default_apply_borrows` pinning output-position
  vars of stuck constraint chains to owned at group quiescence; the
  borrowed-receiver method for-source now compiles and runs balanced on
  both engines.

## Context

Two field bugs, both surfacing as `vm: double free` at runtime, came out of
one afternoon of running a straightforward fs walker:

```talk
func main() {
	walk(Directory(path: Path(["."]))) { entry in print(entry) }
}

func walk(directory: &fs::Directory, fn: (&DirectoryEntry) -> ()) {
	for entry in directory.entries() { fn(entry) }
}
```

**Bug A — a trailing-closure body swallows the caller's temp teardown.**
The MIR builder tracks freshly minted temps in a builder-level accumulator
(`pending_temp_drops` in `src/lower/mir.rs`), drained into `TemporaryEnd`
drop candidates "when the enclosing statement completes" — concretely, by
`drain_temp_drops(current)` calls sprinkled after each node in
`lower_nodes` and at early-exit statements. A call with a trailing block
lowers the closure body *inline, mid-statement*: the `Directory(...)`
receiver temp is still pending when the nested `lower_nodes` walks the
closure body, so the body's first statement-end drain deposits the
caller's `TemporaryEnd` candidate **inside the closure body block**. The
teardown (deinit of `Path.components` + free) then runs once per closure
invocation; the second directory entry frees the same buffers again.
Signature: exactly two entries print, then `vm: double free`.

**Bug B — an owned drop for a value that was never owned.** With a
same-module method on a borrowed receiver as the loop source, the binder
loses its borrow during typing:

```talk
struct D {
	let name: String
	func make() -> Array<Array<String>> { [["x"]] }
}
func walk(d: &D, fn: (&Array<String>) -> ()) {
	for item in d.make() { fn(item) }   // hover: item: Array<String>, owned, needs drop
}
```

`ArrayIterator.next()` returns `Optional<&Element>` (core/Iterable.tlk),
so `item` must be `&Element`; with a *free-function* source it is, and
with the cross-module fs source it is. Flow trusts the type, schedules a
scope-exit deep-drop of the binder each iteration, and lowering emits it —
but the binder was bound to the raw `Optional` payload with no retain, so
the source array's element-drop thunk frees every element a second time
at loop exit. (Flat literal-string payloads mask this: their ByteStorage
is static, so the double free goes undetected.)

The bugs are different, but they are generated by the same two decisions:

1. **Drop placement is a property of builder traversal order, not of
   program structure.** "Which temps are pending" is ambient mutable
   state; "where they drain" is wherever the walk is standing when a
   drain point fires. Every construct that lowers an embedded body
   inline (trailing closures, handler bodies) must remember to isolate
   every piece of statement-scoped state by hand. The builder already
   shows the pattern: `loop_stack` gets a manual `std::mem::take` at
   both embedded-body sites, `handler_stack` is hand-pushed/popped, and
   `pending_temp_drops`, `continuation_temps`, `join_depth`,
   `root_tail_is_return` get nothing. That is a state × sites matrix
   maintained by convention; bug A is the cell that got missed, and each
   new embedded-body construct adds a fresh row.

2. **Nothing verifies the pipeline's output invariant.** The invariant —
   every owned allocation is freed exactly once per execution path — is
   held by a trust chain: flow trusts the types, lowering "emits exactly
   what these schedules say — no re-analysis downstream"
   (`src/flow/drops.rs`), and the only detector is the VM allocator, at
   runtime, in whichever user program happens to trip it. Bug B rode
   that chain end to end: a typing bug became memory corruption with no
   stage in a position to notice.

## Decision

Three changes, each deleting a mechanism rather than patching around it,
plus one contained bug hunt.

**1. A free-balance verifier over the lowered output, on in tests.**
Per-function abstract interpretation of the final λ_G: every `free` /
`deinit` must be justified by exactly one preceding ownership event
(alloc, record/array construction, retain, move-in) on that path, and
every owned production must be consumed exactly once — unconsumed is a
leak, doubly consumed is a double free, both reported with the source
node of the offending drop. Borrows are non-consuming uses. The verifier
runs on the artifact the VM executes, so it back-stops *every* upstream
stage — typing, flow, and lowering — and it is the compile-time successor
to the deleted runtime leak fence: both of today's bugs become red tests
in the existing suite instead of field reports.

Scope for v1: intra-procedural over function bodies; calls consume
arguments per the callee's signature/θ; place-tracking only as deep as
the `free(get_field(...))` chains lowering actually emits. Where aliasing
defeats precision, the verifier stays silent rather than guessing — its
job is catching compiler-emitted imbalance, not user-program linearity
(flow already owns that).

**2. Temp drops become structural; `pending_temp_drops` is deleted.** A
temp minted while lowering statement S belongs to S — Talk's temporary
scope is always the enclosing full statement (we have no lifetime
extension), so the statement lowering itself emits its `TemporaryEnd`
candidates at its own completion block: after the `Call`/`Perform`
statement, which for a call with a trailing block is the join block —
exactly where the `Directory` teardown belongs, after the callee returns.
The statement variants already carry their `temp`; there is no
accumulator, no drain points, and nothing that *can* leak across a body
boundary. This fixes bug A by construction.

**3. One way into an embedded body.** The statement/body-scoped builder
fields (`loop_stack`, `handler_stack`, `continuation_temps`,
`join_depth`, `root_tail_is_return`) move into a single context struct,
and a single `lower_embedded_body()` helper swaps it wholesale. The two
existing sites (handler bodies, trailing blocks) collapse into it; the
per-field `mem::take` discipline is deleted. A future embedded-body
construct cannot forget a field because there is no per-field handling
left to forget.

**4. Hunt bug B's typing root separately.** The `&` on the projected
`Element` is lost only for same-module borrowed-receiver method-call
sources — the cross-module fs path types correctly, which points at a
projection-normalization divergence between the two instantiation paths.
With the verifier in place this is a contained red test, not a crash
report; but the verifier is a net, not the fix, and flow should
additionally refuse to schedule an owned drop for a binder whose
provenance is a borrow projection with no interposed clone (the existing
arm-binder alias rule, made authoritative over the type).

## Stages (suite green after each)

1. **Repro tests (red).** Both shapes above as `vm_tests` on both
   engines: the trailing-closure fs shape (bug A) and the
   borrowed-receiver method-source loop (bug B). They currently crash
   the VM; they start red.
2. **Verifier.** Implement the free-balance pass over lowered bodies;
   wire it into the test drivers so every existing eval test also
   verifies. Assert it flags both repros at compile time — the runtime
   crashes become verifier diagnostics. Expect it to surface additional
   latent imbalance in the suite; triage before proceeding (per the
   stop-when-estimates-break rule).
3. **Structural temp drops.** Statements emit their own `TemporaryEnd`
   candidates; delete `note_pending_temp`, `drain_temp_drops`, the
   `pending_temp_drops` field, and all drain call sites. Repro A goes
   green.
4. **Embedded-body entry point.** Bundle the scoped state, collapse both
   sites into `lower_embedded_body()`, delete the manual takes.
   Behavior-neutral; the diff is negative.
5. **Bug B typing fix.** Find where the projection loses the borrow;
   repro B goes green; add the flow-side provenance guard.

## What this deletes

- `Builder::pending_temp_drops`, `note_pending_temp`,
  `drain_temp_drops`, and every drain call site in `lower_nodes` /
  early-exit statements.
- The per-field `std::mem::take(&mut self.loop_stack)` stash/restore at
  the handler-body and trailing-block sites, and the hand-rolled
  `handler_stack` push/pop discipline around embedded bodies.
- The class of runtime-only detection for compiler-emitted drop bugs:
  after stage 2, an imbalanced lowering fails `cargo test`, not a user's
  program.

## Citations (per decision)

- **Temporary drop points as structural facts.** Rust defines a
  temporary's scope syntactically — "the temporary scope of an
  expression is the smallest scope that contains the expression," drawn
  from a fixed structural list (a statement, a loop body, …) — computed
  before MIR building, not discovered during it:
  [Rust Reference: temporary scopes](https://doc.rust-lang.org/reference/destructors.html#temporary-scopes).
  rustc's MIR builder then schedules drops onto scope frames that unwind
  at scope exit, and refines them with maybe-init dataflow —
  [rustc dev guide: drop elaboration](https://rustc-dev-guide.rust-lang.org/mir/drop-elaboration.html),
  [RFC 1211: MIR](https://rust-lang.github.io/rfcs/1211-mir.html). Talk
  already has the elaboration half (`DropElaboration::{Static, Dead,
  Conditional, Open}` mirrors rustc's categories, per ADR 0010); this
  ADR adopts the placement half.
- **Verifying the generated IR's ownership balance.** Swift SIL in
  Ownership SSA form is statically verified: "values that are not
  consumed along a path" flag as leaks and "double consumes" as
  use-after-frees, explicitly to catch SILGen and optimizer bugs at
  compile time rather than as user-visible memory corruption —
  [Swift SIL Ownership docs](https://github.com/swiftlang/swift/blob/main/docs/SIL/Ownership.md),
  [SIL Ownership Model proposal](https://forums.swift.org/t/sil-ownership-model-proposal-refreshed/16872).
- **Single-entry scoped state over per-field convention.** Standard
  RAII/guard reasoning; the in-repo precedent is ADR 0010's own lesson —
  the same fact computed in two places (flow's drop candidates vs
  lowering's drop stack) disagreed until one authority was named.

## Departures from precedent (deliberate)

- **We verify after lowering, not on an ownership-typed IR.** OSSA
  verifies SIL *before* ownership is lowered away, because SIL carries
  ownership annotations; λ_G carries none, so our verifier reconstructs
  balance abstractly over the final CPS output. Weaker per-value
  precision, but it checks the artifact that actually executes, and it
  needs no new IR.
- **No region scope tree.** Rust computes a temporary-scope tree as a
  separate pass; Talk's rule is degenerate (temp scope = enclosing full
  statement, no lifetime extension), so the statement lowering emitting
  its own candidates *is* the structural rule, with no new pass. If
  temporary lifetime extension ever lands, this decision gets revisited.
