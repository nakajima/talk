# 0012 — Characters are extended grapheme clusters

Status: implemented (2026-07-03)

(0011 is reserved by the dynamic-extent effect handlers ADR, in flight on
the effects branch.)

## Context

`String` was a UTF-8 byte buffer with a byte-level API wearing
character-level names: `length` counted bytes (`"é".length == 2`),
`slice` took byte offsets and could split a multi-byte character, and no
character iteration existed at all. Correct for machines, a lie for
users.

## Decision

talk works with characters; the byte API is demoted to an explicit view.

- **`Character` is an extended grapheme cluster** (UAX #29, Unicode
  17.0.0) — the user-perceived character, the Swift model. `"e\u{301}"`,
  a family-emoji ZWJ sequence, and a regional-indicator pair each count
  as one.
- **Iteration-first**: `String` and `Substring` are `Iterable` over
  `Character`; `count()` is a documented O(n) scan. There is no integer
  character indexing — no `s[i]`, no hidden linear cost.
- **The byte API lives behind `utf8()`**: a borrowed `UTF8View` with
  `at`, `count()`, and `slice` (byte offsets, may split characters —
  that is the point of asking for bytes). The `length` field is renamed
  `byte_count` so nothing character-shaped reads like a character count.
- **`find`/`find_from` return `Int?`** (byte offsets, valid `utf8()`
  offsets) instead of a `-1` sentinel. Byte-wise search stays exact for
  UTF-8: a needle can only match at character boundaries of itself.
- **Equality stays byte equality.** NFC `"é"` and NFD `"é"` compare
  unequal; canonical equivalence needs normalization tables we do not
  carry. Documented at the top of `core/String.tlk`. (Departure from
  Swift, which compares canonically.)
- **`Character` is a borrowed view** (`storage`/`start`/`byte_count`,
  `Borrowed`-marked), like `Substring` — iteration allocates nothing.
  `to_string()` is the owned escape hatch. (Departure from Swift's owned
  `Character` value; talk has first-class borrows.)

## Self-hosted segmentation

The runtime stays bytes-only. Everything Unicode lives in talk source:

- `core/Unicode.tlk` (hand-written): a UTF-8 decoder returning packed
  `scalar * 8 + size` Ints, the UAX #29 rule engine (GB3–GB999,
  including GB9c Indic conjuncts, GB11 emoji ZWJ sequences, GB12/13
  regional-indicator pairs), and the `Character`/iterator layer. Because
  every scan starts at a true boundary, all rule state is local to one
  cluster scan; the iterator carries only a byte offset.
- `core/UnicodeData.tlk` (generated by `cargo run --bin gen_unicode`
  from vendored UCD files, `dev/ucd/17.0.0/`): the break-category table
  as a sorted boundary list packed into one 7-bit-clean string literal —
  four base-128 septets per entry encoding `codepoint * 32 + category`
  (~2,200 entries, ~8.7 KB). A string literal is the only zero-cost bulk
  data vehicle: literals intern as static bytes, while array literals
  lower to per-element runtime stores; and since literals must be valid
  UTF-8, every table byte stays ≤ 0x7F. Lexicographic order equals
  numeric order, so lookup is a binary search over the raw bytes.
- Ill-formed UTF-8 decodes as U+FFFD error units consuming the maximal
  subpart (Unicode §3.9.6, the `from_utf8_lossy` policy) for
  classification only — `Character`s still view the raw bytes, so
  iterating any byte string and concatenating the pieces reproduces it
  exactly.

The compiler grew exactly one instruction for this: `btoi` (Byte → Int
zero-extend, `Byte._toInt()`), without which talk source cannot do
arithmetic on bytes.

## Verification

- `tests/unicode/grapheme_conformance.tlk` (generated): all 766 official
  GraphemeBreakTest-17.0.0 cases (`tests/unicode.rs`), asserting zero
  failures. VM only: the reference evaluator's substitution semantics
  (~100x the VM) put this CPU-bound program at 10-20+ minutes; the
  evaluator covers the same decode/category/boundary code differentially
  through the graphemes corpus and unit tests.
- `tests/programs/graphemes.tlk` + `examples/Graphemes.tlk`: the
  semantic corpus (NFD combining marks, skin-tone modifiers, ZWJ
  families, flags and double flags, Devanagari conjuncts, CRLF,
  decomposed jamo, lossless round-trip) on both engines, with frozen
  stdout for the example.
- `_utf8_decode` and `_grapheme_category` have direct differential
  tests in `src/vm/vm_tests.rs` covering every encoded length, the
  surrogate-gap and range boundaries, truncations, and table endpoints.

## Consequences

- `length`/`byte_at`/`slice` are gone from `String`/`Substring`; core
  and examples migrated. User-facing positional work is iteration,
  `find`, and `utf8()` when bytes are truly meant.
- Unicode upgrades are a data refresh: drop new UCD files under
  `dev/ucd/<version>/`, bump the version pin, regenerate, review the
  diff (`dev/ucd/README.md`).
- `count()` is O(n); anything that needs repeated positional access
  should iterate once and keep what it needs.
- Grapheme classification of *invalid* bytes treats each error unit as
  U+FFFD (category Other); combining marks after an error unit attach
  to it, which is the standard replacement-character reading.
