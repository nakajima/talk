# ADR 0009: Standalone Executables via Bundled Bytecode VM

Status: accepted

Date: 2026-06-24

## Context

Talk currently executes programs by running the compiler pipeline in the `talk`
CLI and immediately interpreting the scheduled VM module:

```text
source -> parse -> resolve names -> type check -> lower to lambda-G -> schedule to vm::Module -> vm::interp::run
```

The VM already has the runtime shape needed for standalone execution:

- `src/vm/mod.rs` defines the register bytecode instruction set and the
  in-memory `vm::Module` representation;
- `src/vm/schedule.rs` lowers lambda-G into VM chunks, registers, constants,
  statics, switch pools, and trap strings;
- `src/vm/interp.rs` executes a `vm::Module` against an `IO` implementation;
- `src/vm/io.rs` defines the host IO boundary.

The missing piece is packaging. Today a user needs the `talk` compiler at run
time because `talk run` compiles from source every time. For distribution, we
want a normal platform executable that can be copied and run without source
files or the compiler.

Native machine-code generation is intentionally out of scope for this decision.
Talk's current backend is the register bytecode VM. The fastest correct path to
standalone executables is to serialize the scheduled VM module and bundle it
with a small runtime that runs the VM.

The user has explicitly accepted requiring a host C compiler/linker (`cc`) when
building standalone executables. This lets `talk build` produce native platform
executables without requiring Cargo or Rust to be installed on the user's system
at build time.

## Decision

Talk will support standalone executables by bundling serialized VM bytecode with
a statically linked Talk runtime.

The primary command will be:

```sh
talk build input.tlk -o output
```

This command will:

1. compile `input.tlk` through the existing frontend, lowering, and scheduler;
2. serialize the resulting `vm::Module` into a versioned Talk bytecode image;
3. generate a small C launcher that embeds the bytecode image as a static byte
   array;
4. invoke `cc` to compile and link that launcher with a Talk runtime static
   library;
5. write a platform executable at `output`.

Conceptually:

```text
input.tlk
  -> talk compiler
  -> vm::Module
  -> Talk bytecode image
  -> generated C launcher
  -> cc launcher.c libtalk_runtime.a -o output
```

The generated C launcher is deliberately tiny:

```c
#include <stdint.h>
#include <stddef.h>

extern int talk_runtime_run(const uint8_t *bytes, size_t len);

static const uint8_t TALK_PROGRAM[] = {
    /* generated bytecode bytes */
};

int main(int argc, char **argv) {
    (void)argc;
    (void)argv;
    return talk_runtime_run(TALK_PROGRAM, sizeof(TALK_PROGRAM));
}
```

The Rust runtime exposes a C ABI entry point:

```rust
#[unsafe(no_mangle)]
pub extern "C" fn talk_runtime_run(ptr: *const u8, len: usize) -> i32 {
    // validate pointer/length
    // decode Talk bytecode into vm::Module
    // run vm::interp::run with StdioIO
    // render diagnostics to stderr
    // return a process exit code
}
```

The runtime entry point owns bytecode decoding and execution. The generated C
file only embeds bytes and calls the runtime.

## Non-Goals

This ADR does not introduce native code generation. The standalone executable is
native only in the packaging sense: it is a platform executable containing the
Talk VM plus bytecode.

This ADR does not require cross-compilation support. `talk build` targets the
host platform through the host `cc` by default. Cross-compilation can be added
later by allowing users to choose a target-specific runtime archive and compiler
command.

This ADR does not define a stable public ABI for arbitrary C callers beyond the
minimal runtime entry point needed by the generated launcher.

This ADR does not require a compact or optimized bytecode encoding in the first
implementation. The format should be explicit and versioned, but size and load
speed are secondary to correctness and validation.

## Bytecode Image Format

Talk will add an explicit serialized representation for `vm::Module`.

The image must begin with enough metadata to reject incompatible inputs before
interpretation:

```text
magic:        "TALKBC\0"
format_ver:   u32
runtime_min:  u32, optional in v1 if crate version is enough
flags:        u32
payload:      encoded module sections
```

The payload encodes the full scheduled module:

- entry chunk index;
- chunks: name, arity, register count, instruction list;
- constants;
- argument pool;
- switch target pool;
- trap strings;
- static memory image.

Instruction encoding is by explicit opcode tags, not by dumping Rust enum memory
or relying on derived enum layout. This keeps the bytecode format independent of
Rust's in-memory representation.

The decoder must validate:

- magic and version;
- section lengths and integer overflow;
- UTF-8 strings where strings are expected;
- chunk indices referenced by calls and closures;
- register indices against each chunk's register count;
- constant, argument-pool, switch-pool, and trap indices;
- switch target ranges;
- entry chunk existence.

Invalid bytecode must produce a runtime error, not a panic or memory-unsafe
behavior.

For v1, bytecode is a private Talk implementation format. Compatibility is only
required between a `talk` compiler and the `libtalk_runtime.a` shipped with the
same Talk release. The header version exists so mismatches fail clearly.

## Constants and Runtime Values

The bytecode constant pool should initially support only values that are valid
compile-time constants for scheduled modules:

- integers;
- floats encoded by bits;
- booleans;
- bytes;
- void;
- static pointers into the module static memory image.

Runtime-only values such as records, tuples, variants, existentials, closures,
and cells should not appear in the serialized constant pool unless a later
compiler change intentionally permits them and defines their encoding. If the
encoder encounters an unsupported constant, compilation to bytecode must fail
with an internal compiler error or explicit unsupported-bytecode diagnostic.

## Runtime Packaging

A Talk runtime static library will contain:

- bytecode decoder;
- `vm::Module`, `Chunk`, `Insn`, `MemKind`, and `IoOp` definitions needed by
  the decoder and interpreter;
- VM interpreter;
- host `StdioIO` implementation;
- C ABI entry point `talk_runtime_run`.

Initially this may live in the existing `talk` crate as an additional staticlib
output if that is the smallest implementation. The desired long-term shape is a
separate runtime crate:

```text
talk           compiler, CLI, LSP, REPL, frontend, lowerer, scheduler
talk-runtime   bytecode decoder, VM, IO runtime, C ABI
```

The split is not required before `talk build` lands. It is a follow-up to reduce
linked code size and make ownership boundaries clear.

The `talk` CLI must know how to locate the runtime archive. Acceptable v1
strategies are:

- runtime archive installed next to the `talk` executable;
- runtime archive path provided at build time and embedded in the CLI;
- explicit `--runtime /path/to/libtalk_runtime.a` override for development.

## CLI Surface

The standalone-executable path adds these commands or flags:

```sh
talk build input.tlk -o output
```

Build a standalone executable.

```sh
talk build --emit-bytecode input.tlk -o output.tbc
```

Compile source to a bytecode image without linking an executable. This is useful
for tests, debugging, and future tooling.

```sh
talk run-bytecode output.tbc
```

Run a bytecode image through the runtime path. This gives the bytecode decoder
and VM runner a direct test surface before C launcher/linking work is involved.

The existing command remains:

```sh
talk run input.tlk
```

`talk run` continues to compile source and execute the in-memory scheduled
module. It is the development path. `talk build` is the distribution path.

## Error and Exit Semantics

For v1, standalone executables should match `talk run` behavior as closely as
possible:

- if the program returns `Void`, print nothing extra;
- if the program returns a non-void value, print the debug-style value in the
  same way `talk run` currently does;
- program stdout goes to host stdout through `StdioIO`;
- VM/runtime errors are written to stderr;
- runtime errors exit nonzero.

Compilation failures happen during `talk build`, not at executable startup.
Startup failures should be limited to bytecode/runtime version mismatches,
corrupt embedded bytecode, or VM runtime errors.

Future work can introduce source-level process exit APIs and argument handling.
For v1, `argc` and `argv` are ignored by the generated launcher.

## Implementation Plan

### Stage 1: Bytecode encode/decode

Add a bytecode module, likely under `src/vm/bytecode.rs`, with:

- `encode_module(&vm::Module) -> Result<Vec<u8>, EncodeError>`;
- `decode_module(&[u8]) -> Result<vm::Module, DecodeError>`.

The encoder and decoder use explicit opcode tags and section encodings. Tests
round-trip scheduled modules and compare the original and decoded module by
running both.

### Stage 2: CLI bytecode commands

Add:

- `talk build --emit-bytecode input.tlk -o output.tbc`;
- `talk run-bytecode output.tbc`.

Tests compare these paths:

```text
talk run input.tlk
talk build --emit-bytecode input.tlk -o input.tbc
talk run-bytecode input.tbc
```

The observable stdout and process result should match.

### Stage 3: Runtime C ABI

Expose:

```rust
#[unsafe(no_mangle)]
pub extern "C" fn talk_runtime_run(ptr: *const u8, len: usize) -> i32
```

This function validates the input pointer, decodes the bytecode image, runs the
VM with `StdioIO`, reports errors, and returns an exit code.

For safety, the FFI boundary must not unwind across C. Any internal panic should
be caught or prevented. Normal decode and VM failures should return nonzero.

### Stage 4: Runtime static library

Build or package a `libtalk_runtime.a` compatible with the current compiler.

Initially, this can be produced from the main crate if that is simplest. Later,
move the runtime code into a dedicated crate so standalone programs do not link
frontend/compiler/LSP code.

### Stage 5: Generated C launcher and `cc` invocation

Add `talk build input.tlk -o output`:

1. compile to bytecode bytes;
2. write a temporary C file embedding those bytes;
3. locate the runtime archive;
4. invoke `cc` with the generated C file and runtime archive;
5. write the executable at `output`;
6. clean temporary files unless `--keep-temps` is provided.

A useful debug flag:

```sh
talk build input.tlk -o output --keep-temps
```

This leaves the generated launcher and bytecode file on disk for inspection.

## Consequences

Standalone executable support reuses the existing backend and VM rather than
starting a native backend. That keeps the feature aligned with the current
correctness story: the scheduler and VM remain the only execution backend that
must match the reference evaluator.

The first executables will be larger and slower than native code because they
contain an interpreter. This is acceptable. The goal is distribution and
operational convenience, not peak performance.

Requiring `cc` keeps Talk from depending on Cargo/Rust during per-program builds.
It does mean users need platform C toolchain support installed. This is an
explicitly accepted tradeoff.

A bytecode format becomes part of the backend. Even if private in v1, it creates
a new compatibility seam. Versioning and validation are mandatory from the
start.

A runtime crate split becomes more valuable after this feature lands. Without a
split, standalone executables may link more code than necessary. That is a size
problem, not a correctness blocker.

## Risks

### Rust enum drift

`Insn`, `Value`, and related runtime types will continue to evolve. The bytecode
format must not serialize their memory layout or rely on derived discriminants.
Explicit opcode tags and format-version tests reduce this risk.

### Runtime/compiler mismatch

A compiler may emit bytecode the runtime does not understand. The bytecode
header and format version must reject mismatches before interpretation.

### Incomplete constant encoding

If future scheduling starts placing richer runtime values in the constant pool,
the encoder may reject modules that `talk run` can execute in memory. This is
acceptable: the fix is to intentionally extend the bytecode format, not to
silently serialize unsupported values.

### Linker portability

`cc` behavior differs across platforms. V1 should target Unix-like hosts first,
because the current `StdioIO` uses POSIX/libc behavior. Windows support can be
added after the runtime IO layer has a Windows implementation or a compatibility
strategy.

### Panic across FFI

The runtime entry point must not unwind into C. The decoder and VM should return
errors normally, and the FFI wrapper should prevent unwinding across the ABI
boundary.

## Alternatives Considered

### Generate a Rust runner and invoke Cargo

This is simpler to prototype: generate a tiny Rust crate using `include_bytes!`
and call `cargo build --release`. It requires Rust and Cargo for every Talk
program build. Since requiring `cc` is acceptable, the C launcher plus static
runtime is a better distribution model.

### Append bytecode to a prebuilt runner executable

This avoids `cc` entirely. `talk build` would copy a platform runner executable,
append bytecode, and patch a footer with the bytecode offset and length. This is
a good future optimization, but it requires per-platform runner binaries and
self-introspection logic. The C launcher is more transparent and easier to debug.

### Interpret source at startup

The executable could embed source and the full compiler, then compile on launch.
That preserves behavior but defeats the purpose: startup is slower, errors move
to runtime, and the executable must include the compiler.

### Native code generation

Native codegen could eventually produce faster and smaller binaries. It would
require a new backend, calling convention, runtime representation decisions,
closure lowering, IO ABI, and careful interaction with effects and ownership.
That is much larger than standalone packaging and is not needed for the first
working executable story.

## Validation

The feature is complete when:

1. bytecode encode/decode round-trips scheduled modules used by the test suite;
2. `talk run-bytecode` matches `talk run` for representative examples;
3. `talk build example.tlk -o example` produces an executable;
4. running the executable matches `talk run example.tlk` for stdout and exit
   behavior;
5. corrupt bytecode and version mismatch tests fail cleanly;
6. CI covers at least the bytecode path and, where a C toolchain is available,
   the generated executable path.

The existing VM/reference-evaluator agreement remains the semantic baseline.
Standalone packaging must not become a second semantics.
